## What Went Well

### 1. Module boundary discipline held throughout

`import requests` appeared only in `vapix.py`. `print()` appeared only in `reporter.py`. `config.timeout` was never hardcoded. These constraints were declared in Story 3.1 dev notes and not violated once across all 5 stories. The architecture's layered responsibility model worked exactly as designed.

### 2. Learnings propagated through story files, not just PR comments

When code review caught a pattern flaw — the fixture mutation bug in Story 3.1, the vacuous-loop pattern in 3.2 and 3.3 — the fix was added to the next story's "Previous Story Learnings" section. This meant the next implementor had the lesson in context, not buried in a PR thread. The fixture copy pattern (`{**fixture, key: val}`) and `mock_set.assert_not_called()` became permanent conventions with zero subsequent recurrence.

### 3. Code review did real work on boundary stories

Stories 3.4 and 3.5 had the highest patch counts (6 and 5 respectively) because they wired the module boundaries together. Reviews caught: missing argument assertions on mock calls, inline imports that hide ImportError until runtime, missing credentials checks in executor tests, and a silent `else` clause that would swallow unknown enum values. These were not trivial catches.

### 4. The executor failure-isolation model is clean

`apply_all()` is 15 lines and never raises. Its contract — catch any `Exception`, convert to `CameraResult(FAILED)`, continue — is fully tested across 7 scenarios including bare `Exception`, `VapixError`, and empty fleet. Story 3.5 wired the exit-code contract correctly: reporter returns `int`, CLI raises `typer.Exit(code=exit_code)`.

## What Needs Improvement

### 1. VAPIX parameter names are entirely unverified — this is the top risk

All 7 VAPIX constants in `reconciler.py` are marked `# UNVERIFIED`:

```python
_SMB_GROUP      = "root.Network.Share"        # UNVERIFIED
_SMB_HOST       = "root.Network.Share.Host"   # UNVERIFIED
_SMB_SHARE      = "root.Network.Share.Share"  # UNVERIFIED
_SMB_USER       = "root.Network.Share.Username" # UNVERIFIED
_SMB_PASS       = "root.Network.Share.Password" # UNVERIFIED
_MOTION_ENABLED = "root.Motion.M0.Enabled"    # UNVERIFIED
_MOTION_SENSITIVITY = "root.Motion.M0.Sensitivity" # UNVERIFIED
```

The tool passes 77 tests and is functionally complete against mocks. Against real hardware with wrong param names: every `get_params` call returns `None` for all keys → `None != expected` → every setting is written on every run → idempotency is broken silently.

**This must be resolved before any production use.**

### 2. Deferred work is now 28 items — some are hardware-verification-blocked, some are not

Hardware-blocked (cannot fix until VAPIX params verified):
- `dict.get()` returns None on absent key → spurious SET on factory-reset cameras
- Non-canonical numeric string from camera (`"050"` vs `"50"`) → permanent spurious SET loop
- `_MOTION_GROUP` and `M0` window index need confirmation

Not hardware-blocked (code-level gaps):
- `motion_sensitivity` float slip-through: `str(50.0)` → `"50.0"` → permanent diff vs camera's `"50"` — add `int(config.motion_sensitivity)` cast in `reconciler.py`
- `APPLIED` with empty `settings_changed` renders as `applied ()` — add `if settings_changed` guard
- `FAILED` with `error=None` renders as `"FAILED — None"` — add `result.error or "unknown error"` guard
- `apply` exits 0 when scanner returns no cameras — `list` exits 2; apply should too
- `apply --subnet` override path untested for apply (equivalent test exists for list)

### 3. Vacuous assertion pattern appeared in 3 consecutive stories

Tests that iterate an empty `call_args_list` always pass regardless of actual behavior. Found in Stories 3.1, 3.2, and 3.3. All three were patched in review. The fix (`mock.assert_not_called()`) is now in learnings, but this pattern appearing in 3/5 stories suggests it should be a checklist item during development, not review.

## Key Technical Decisions to Carry Forward

| Decision | Rationale | Status |
|----------|-----------|--------|
| `catch Exception` in executor, not just `VapixError` | Executor is full failure-isolation boundary; unexpected exceptions must not abort the fleet loop | Keep |
| Motion enabled + sensitivity grouped in one SET call | Avoids partial-apply: never update sensitivity without confirming enabled state | Keep |
| `smb_ip` and `smb_creds` as separate SET calls + labels | Independent failure modes; IP can succeed while creds fail | Keep |
| VAPIX param name constants isolated in `reconciler.py` | Single file to update when hardware verification reveals correct names | Keep — update constants in-place during hardware verification pass |
| `reporter.py` returns exit code as `int`; `cli.py` raises `typer.Exit` | Clean separation: reporter is pure output, CLI owns process lifecycle | Keep |

## Retrospective on Process

**What to keep doing:**
- Carry learnings explicitly into next story's dev notes — it works
- Three-layer code review (blind hunt, edge cases, AC audit) — found real bugs every story
- Deferred work logged with specific locations — makes hardening planning concrete

**What to start doing:**
- Add "is any `call_args_list` loop vacuous?" to dev checklist — not just review checklist
- Verify VAPIX constants on hardware *before* writing tests that pin on them — saves a round-trip of updating both code and tests

**What to stop doing:**
- Deferring hardware verification beyond the story that introduces the constants — the risk compounds each story that builds on unverified param names
